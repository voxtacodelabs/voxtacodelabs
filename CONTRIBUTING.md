# Contribuindo para a Voxta CodeLabs

Primeiramente, obrigado pelo seu interesse em contribuir com os projetos da **Voxta CodeLabs Tecnologia Ltda.**

Acreditamos que a inovação acontece por meio da colaboração. Este documento descreve as diretrizes para garantir que todas as contribuições mantenham o padrão de qualidade, organização e excelência adotado pela empresa.

---

# Nossa Filosofia

Na Voxta CodeLabs, não desenvolvemos apenas software.

Desenvolvemos soluções inteligentes, escaláveis e sustentáveis que unem:

- Inteligência Artificial
- Engenharia de Software
- Ciência de Dados
- Cloud Computing
- Automação
- Engenharia de Plataforma

Cada contribuição deve refletir esses princípios.

---

# Antes de contribuir

Antes de iniciar qualquer desenvolvimento:

- Leia completamente o README.
- Consulte o ROADMAP do projeto.
- Verifique se já existe uma Issue relacionada.
- Abra uma Discussion quando necessário.
- Certifique-se de compreender a arquitetura do projeto.

---

# Fluxo de Desenvolvimento

Seguimos uma adaptação do **GitHub Flow**.

## 1. Fork

Realize um Fork do projeto.

## 2. Clone

```bash
git clone https://github.com/SEU-USUARIO/repositorio.git
```

## 3. Crie uma Branch

Utilize nomes claros.

Exemplos:

```text
feature/login-api

feature/dashboard

bugfix/token-expiration

docs/readme-update

refactor/cache

hotfix/security
```

---

# Padrão de Branches

| Prefixo | Uso |
|----------|------------------------------|
| feature/ | Nova funcionalidade |
| bugfix/ | Correção de bugs |
| hotfix/ | Correções críticas |
| refactor/ | Refatoração |
| docs/ | Documentação |
| test/ | Testes |
| ci/ | CI/CD |
| chore/ | Tarefas administrativas |

---

# Commits

Seguimos o padrão **Conventional Commits**.

## Exemplos

```text
feat(auth): adiciona autenticação JWT

fix(api): corrige timeout

docs(readme): atualiza documentação

refactor(core): melhora arquitetura

test(user): adiciona testes unitários

chore(deps): atualiza dependências
```

---

# Pull Requests

Todo Pull Request deve conter:

- Objetivo da alteração
- Motivação
- Issue relacionada
- Evidências de testes
- Capturas de tela (quando aplicável)
- Impactos conhecidos

---

# Checklist

Antes de enviar um Pull Request confirme:

- Código compilando
- Testes executados
- Sem conflitos
- Documentação atualizada
- Código comentado quando necessário

---

# Padrão de Código

Priorizamos:

- Legibilidade
- Simplicidade
- Modularização
- Reutilização
- Performance
- Segurança

Evite duplicação de código.

---

# Documentação

Toda funcionalidade relevante deve possuir documentação.

Sempre que possível incluir:

- Diagramas
- Fluxogramas
- Exemplos
- Casos de uso
- Limitações

---

# Testes

Sempre escreva testes quando possível.

Tipos recomendados:

- Unitários
- Integração
- End-to-End
- Performance

---

# Segurança

Nunca envie:

- Senhas
- Tokens
- Chaves privadas
- Credenciais
- Dados sensíveis

Utilize sempre arquivos `.env`.

---

# Dependências

Adicione apenas bibliotecas realmente necessárias.

Antes de adicionar uma nova dependência avalie:

- Manutenção
- Licença
- Segurança
- Comunidade
- Performance

---

# Inteligência Artificial

Projetos envolvendo IA devem priorizar:

- Reprodutibilidade
- Transparência
- Ética
- Explicabilidade
- Segurança

Modelos devem possuir documentação.

---
